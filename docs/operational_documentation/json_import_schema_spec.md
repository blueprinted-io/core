# Blueprinted JSON Import Schema Specification

Version: 1.0  
Status: draft

---

## Purpose

This document defines the JSON import schema for bulk ingestion of domain content into Blueprinted. It covers the schema structure, field definitions, design decisions, and the LLM prompt contract for generating conformant payloads.

---

## Design Decisions

### 1. Hierarchical payload (domain → workflows → tasks)

The schema accepts a full domain hierarchy in a single payload. Tasks and workflows are defined together but kept as separate top-level arrays. This solves the chicken-and-egg problem of needing workflows to exist before tasks can be attached to them — the importer creates both in a single transaction.

### 2. Temporary import IDs

Tasks are assigned short sequential IDs (`T001`, `T002` etc.) for the purpose of wiring workflow→task relationships within the import payload. These IDs are throwaway scaffolding:

- They are generated by the LLM during document generation
- They are used only by the importer to resolve references
- They are never written to the database
- The database generates its own IDs on record creation

Sequential numeric IDs with a fixed prefix were chosen over natural language title references because they are shorter, opaque, and harder for a model to drift on. The model only needs to maintain consistency within a single generation pass — not across sessions or documents.

### 3. Deduplication at review time, not import time

The schema makes no attempt to deduplicate tasks against existing database records at import time. Automated semantic matching at ingress would require the system to make judgement calls about whether two tasks are "the same thing" — a semantic decision that requires human review.

Instead:

- All imported tasks are created as `draft` or `submitted` (per the `auto_submit_on_import` system setting)
- At review time, the system runs fuzzy matching against confirmed tasks in the same domain
- If candidates above a similarity threshold are found, the reviewer is shown a banner: *"This task may already exist — similar tasks found:"* with links to candidates
- The reviewer decides: confirm the new task, discard it, or incorporate its content into the existing record

This is consistent with Blueprinted's governance model — automated systems surface candidates, humans make decisions. Deduplication quality improves over time as the confirmed task pool grows.

As the confirmed task pool matures, LLM prompts generating future imports can be supplied with existing task titles to reduce duplicate generation at source.

### 4. Imports never create confirmed records

Regardless of the `auto_submit_on_import` setting, imported records are created at `draft` or `submitted` status only. The `confirmed` status can never be set by import. This is a non-negotiable governance rule.

### 5. Workflow confirmation dependency

A workflow can only be confirmed once all referenced task versions are confirmed. Imported workflows are created in `draft` state and remain unconfirmable until their tasks clear the review cycle. This is correct behaviour per the governance model, not a workaround.

## Schema

```json
{
  "schema_version": "1.0",
  "domain": "string",
  "tasks": [
    {
      "id": "T001",
      "title": "string",
      "outcome": "string",
      "facts": ["string"],
      "concepts": ["string"],
      "procedure_name": "string",
      "dependencies": ["string"],
      "irreversible": false,
      "steps": [
        {
          "text": "string",
          "completion": "string",
          "actions": ["string"]
        }
      ]
    }
  ],
  "workflows": [
    {
      "title": "string",
      "objective": "string",
      "task_order": ["T001", "T002"]
    }
  ]
}
```

## Field Definitions

### Top level

| Field | Type | Required | Notes |
|---|---|---|---|
| `schema_version` | string | Yes | Must be `"1.0"`. Used by importer to select correct parsing logic. |
| `domain` | string | Yes | Must match an existing domain in the target Blueprinted instance. |
| `tasks` | array | Yes | Flat list of all task definitions in this payload. Must not be empty. |
| `workflows` | array | Yes | List of workflow definitions referencing tasks by import ID. May be empty if importing tasks only. |

### Task object

| Field | Type | Required | Notes |
|---|---|---|---|
| `id` | string | Yes | Sequential import ID (`T001`, `T002` etc.). Unique within this document. Never written to DB. |
| `title` | string | Yes | Clear, learner-facing task name. Imperative form. |
| `outcome` | string | Yes | Single observable state change produced by completing this task. |
| `facts` | array of strings | Yes | Literal information required to execute the task. May be empty array `[]`. |
| `concepts` | array of strings | Yes | Minimal mental models required to execute the task correctly. May be empty array `[]`. |
| `procedure_name` | string | Yes | Name of the step sequence. |
| `dependencies` | array of strings | Yes | Conditions that must be true before this task can be executed. May be empty array `[]`. |
| `irreversible` | boolean | Yes | `true` if the task produces changes that cannot be undone. Always present, never omitted. |
| `steps` | array of step objects | Yes | Ordered atomic instructions. Must not be empty. |

### Step object

| Field | Type | Required | Notes |
|---|---|---|---|
| `text` | string | Yes | The instruction. Imperative form. Single operation only. Names an explicit object or target. |
| `completion` | string | Yes | Observable condition confirming the step is complete. |
| `actions` | array of strings | No | Optional how-to substeps providing concrete method guidance. May be empty array `[]`. |

### Workflow object

| Field | Type | Required | Notes |
|---|---|---|---|
| `title` | string | Yes | Name of the workflow. |
| `objective` | string | Yes | The measurable outcome the workflow achieves. Org-defined. |
| `task_order` | array of strings | Yes | Ordered list of task import IDs. All IDs must exist in the `tasks` array. Must not be empty. |

## Validation Rules

The importer enforces the following before creating any database records. A failed validation rejects the entire payload with a descriptive error — no partial imports.

| Rule | Error |
|---|---|
| `schema_version` must be `"1.0"` | Unsupported schema version |
| `domain` must exist in the target instance | Unknown domain |
| All task `id` values must be unique within the payload | Duplicate task ID |
| Task ID sequence must follow `T001`, `T002` format | Invalid task ID format |
| All `task_order` references must match a task `id` in the payload | Unresolved task reference |
| No task `steps` array may be empty | Task has no steps |
| No `task_order` array may be empty | Workflow has no tasks |
| `irreversible` must be a boolean, always present | Missing or invalid irreversible field |
| `confirmed` status may not be set by import | Import cannot confirm records |

## Two-Pass Generation

Generating a conformant import payload is a two-pass process. Attempting both passes in a single generation produces lower quality output — the LLM is simultaneously reasoning about structure and writing detailed content, which causes tasks to be too broad and steps to be underspecified.

### Pass 1 — Structure

Identify the workflow and task structure from the source material. For each item, reason from the bottom up:

1. **Find the non-divisible actions first.** These are the procedure steps — actions that require a decision or produce an observable change in system state. Actions that are purely mechanical confirmation of a prior step (e.g. clicking OK after making a selection) are not steps.
2. **The thing immediately above non-divisible steps is a task.** If the steps underneath an item are genuinely atomic and non-divisible, the item is a task regardless of how large it feels.
3. **Look upward for workflow groupings.** Does this task fit within a larger series of tasks that together achieve a single bigger outcome? If yes, it belongs to a workflow. If no natural grouping exists, it is a single-task workflow.

Pass 1 output: task stubs (ID + title + outcome only) and workflow stubs (title + objective + task_order). No steps, facts, concepts, or dependencies yet.

### Pass 2 — Content

Populate each task fully using the workflow context established in Pass 1. Write facts, concepts, dependencies, and steps for each task. Apply the completion-condition test to every step (see below).

Pass 2 output: the complete conformant JSON payload.

## Step Atomicity and the Completion Condition Test

The completion condition field is not just documentation — it is a validity test applied during generation.

**A step is atomic if:**
- It requires the operator to make a judgement or selection, OR
- It produces an observable, independently verifiable change in system state

**A step is NOT atomic if:**
- Its completion condition is a restatement of the action ("you have clicked OK", "the button was pressed")
- It is purely mechanical confirmation of a prior step with no decision or state change

**The generation rule:** write the completion condition first. If the completion condition is circular or describes no observable state change, discard the step and merge its action into the preceding step.

**Examples:**

| Step text | Completion condition | Valid? | Reason |
|---|---|---|---|
| Click OK | You clicked OK | No | Circular — restatement of the action |
| Click Next | The next wizard page is displayed | Maybe | Only valid if the page transition represents a meaningful state change |
| Configure the network interface with the assigned IP address | Network interface shows correct IP; ping to gateway succeeds | Yes | Observable, independently verifiable state change |
| Accept the licence agreement | Licence acceptance is recorded and the installer proceeds to the next stage | Yes | Has a meaningful system consequence |

This test encodes the human definition of atomicity — the smallest step that requires a decision or has a meaningful consequence — and prevents the LLM from decomposing steps to a level of granularity that is technically correct but operationally useless.

## LLM Prompt Contract

When using an LLM to generate import payloads, the following rules must be included in the system prompt. This contract applies to Pass 2 (content generation). Pass 1 (structure) should be run as a separate prompt.

```
You are generating a Blueprinted JSON import payload. Follow these rules exactly:

1. Output valid JSON only. No preamble, no explanation, no markdown code fences.

2. Assign each task a sequential ID starting at T001, incrementing by one (T001, T002, T003...). 
   Do not skip, reuse, or modify IDs.

3. Every field defined in the schema must be present in every object, even if the value 
   is null, false, or an empty array. Never omit a field.

4. When referencing tasks in workflow task_order, copy the task ID exactly as assigned 
   (e.g. "T001"). Do not use task titles in task_order.

5. For every step, write the completion condition first. If the completion condition is 
   a restatement of the action ("you have clicked OK", "the button was pressed"), discard 
   the step and merge its action into the preceding step. Only keep steps where the 
   completion condition describes an observable, independently verifiable state change.

6. Steps must be atomic — one operation per step. No compound instructions. 
   No conditional logic. Imperative form only. A step is atomic if it requires a decision 
   or produces an observable change in system state. Purely mechanical confirmation of a 
   prior step (clicking OK, clicking Next where no decision is required) is not a step.

7. The irreversible field must always be present and must be a boolean (true or false).

8. facts, concepts, dependencies, and actions may be empty arrays ([]) but must always 
   be present.

9. Do not invent domain names. Use only the domain value provided to you.
```

## Importer Behaviour

1. Validate the full payload against all validation rules before creating any records
2. Create all tasks first, capturing DB-generated IDs against import IDs in a local map
3. Create all workflows, resolving `task_order` import IDs to DB task IDs using the local map
4. Apply `auto_submit_on_import` setting to determine created status (`draft` or `submitted`)
5. Discard the import ID map — temp IDs are never stored
6. Return a structured import report: records created, any warnings

## Future Considerations

- **Fuzzy duplicate detection at review time**: when a reviewer opens an imported task, the system checks confirmed tasks in the same domain for semantic similarity and surfaces candidates above a configurable threshold. The reviewer decides whether to confirm, discard, or merge.
- **Existing task pool as prompt context**: once a confirmed task pool exists, future LLM import prompts can be supplied with existing task titles to reduce duplicate generation at source.
- **Idempotent re-import**: if the same payload is submitted twice, detect via SHA-256 hash (consistent with existing ingestion duplicate detection) and redirect to the existing import job.
- **Schema versioning**: the `schema_version` field allows the importer to handle future schema evolution without breaking existing payloads.
